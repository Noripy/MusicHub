クラブやフェスの現場で聴いた音楽を、**イベント単位**で記録できるアプリ。識別できない未リリース曲・Bootleg・Mashupも、曲名が空欄のまま残せる。

### 5-2. それでも自分が作りたい理由

既存サービスがある中で、

- なぜ満足できなかったのか
- どこに違和感があったのか

※「差別化ポイント」ではなく「不満・ズレ」に注目してください。

- **MeloLog**: DJ/Producer名でキーワード検索しても見つからないことがある。Spotify APIでヒットした楽曲のみで、現地で聴いた未リリース曲・Remix・Bootlegは出てこない。

- **EMOMU**: SoundCloud連携を前提としているが、オーディエンスの中にはSoundCloudアカウントを持っていない人がいる。複数プラットフォームを駆使する音楽好きにとって、特定プラットフォーム依存は使いづらい。

- **Shazam**: ドロップ(サビ)の部分で解析するとヒットしやすいが、誤判定や、録音の分解能不足で判定されないまま終わることがある。**識別できなかった時点でサービスとしての役目が終わる**のが最大の限界。

- **Spotify**: リリース済み楽曲はストリーミングできるが、現場でかかる一部のRemix・Bootleg・Mashup・未リリース曲はヒットしない。

- **1001tracklists**: 有名なTop DJ/Producerしか公開トラックリストが共有されることがないため、国内のDJ/Producerの公開トラックリストが載っている訳ではない。

**共通する違和感**:既存サービスはすべて「曲が特定済み」を前提にしている。Shazamで識別できなかった瞬間、EMOMUでSoundCloudにない曲だった瞬間、Spotifyに登録されていないBootlegだった瞬間、ユーザーは記録手段を失う。MusicHubは「自分が現場で出会った未識別の曲を、自分の記憶の手がかりと一緒に残すサービス」として扱う点で、これらすべてと異なる。

## 7. このアプリで実現すること

### MVPで作る機能

- ユーザーのログイン認証(Rails 8 built-in authentication)
- **Eventモデル**で、ユーザーが、いつどこの会場で何のイベントで誰のDJがパフォーマンスしている時の楽曲だったかを記録する
- **TrackEntryモデル**で、Eventモデルに関連した楽曲を、雰囲気（例. 激しい、明るい、落ち着いた etc...）・音楽ジャンル・体感BPMで記録する
- 出会った瞬間の楽曲が分からなくても登録できるように、曲名を空欄で登録できる
- Eventモデルと関連するTrackEntryモデルから、曲名が空欄でも思い出せるように未識別一覧で管理する
- 曲名が判明したら、後から曲名をテキスト形式で追記して更新できる

### 本リリースで作る機能
- ユーザーが体験した音楽イベントをSNSで共有するために、イベント単位の共有リンク発行する
- ユーザーが楽曲ごとにタグ付けして、タグ検索で絞り込み検索することで思い出しやすくする
- ユーザーが、プレイリストを作成し、プレイリストの共有リンク発行
- オーディエンスが多く回線混雑したり会場が電波が届きづらい空間でもユーザーが記録できるようにする

## 8. このアプリの懸念点とその対策
※最低1つは必ず書いてください。

このアプリが**うまくいかない可能性**と、
それに対して**どう向き合おうとしているか**を書いてください。

### 懸念点①

- 何が問題になりそうか：既存サービスとの差別化
- なぜそう思うか：MeloLog/Spotify/EMOMU/Shazamと表面的な機能が被って見える可能性がある。
- そのために考えている対策：
  - 「Event×TrackEntryのデータモデル」と「曲名nullを許容する記録フロー」をMVP段階から実装し、データモデルレベルで競合と異なることを示す
  - Shazamは識別装置・MusicHubは記憶装置という役割分担を明示する(同じ土俵で勝負しない)
  - 「未識別エントリ一覧」=後で答えに行き着くTODOリスト、というShazamに存在しないUXを作る


### 懸念点②

- 何が問題になりそうか：業界・ユーザーの解像度不足
- なぜそう思うか：現場でDJされている方や現場スタッフ、会場に足を運ぶオーディエンスのリサーチが十分にできていない。
- そのために考えている対策：身近な友人(クラブ通いの仲間)から実際のフィードバックを得ながらMVPを改善。本リリース開発開始後、現地調査も並行する。


### 懸念点③

- 何が問題になりそうか：オフライン環境への対応
- なぜそう思うか：クラブ会場やフェス会場によっては、オーディエンスが多く回線混雑したり会場が電波が届きづらい空間の影響で、圏外もしくは通信速度低下しwebアプリが機能しないことがある。
- そのために考えている対策：「会場を出てから記録する」前提で設計する。WA化・IndexedDB・Solid Queueリトライを段階的に導入。MVPで全部やろうとすると間に合わないため、本リリースで導入予定。

  - なるべくコストを抑えつつ運用しやすいアーキテクチャにしたい

- 今回チャレンジしたい点
  - 新しい音楽に触れたくてクラブ・フェス会場に来ているユーザーに向けて、楽曲記録をイベント単位で管理する
  - Rails 8 built-in authenticationを使った認証実装(Rails 8新機能の活用)
  - RSpecを活用したテスト駆動開発
- 不安な点
  - 現場での利便性を考えるとShazaで満足してしまう問題がある。MVPでのShazam差別化フローとして以下のように考えている：

```
現場でShazamが解析失敗 → MusicHubで曲名空欄エントリ作成
                       (イベント・雰囲気タグ・体感BPM・記憶メモを残す)
                            ↓
                  後日、別ルート(動画、知人、tracklists等)で判明
                            ↓
                  MusicHubで該当エントリを編集して曲名を追記
                            ↓
                  「未識別エントリ一覧」から消えて識別済みに移動
```

このフローを実装することで、Shazamではできない「答えが出るまで待ってくれる」サービスになる想定。

### 10-3.キャッチアップ不足の懸念

キャッチアップ不足の段階では以下の懸念点があります。
1. **完成までのハードルが高い**
使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想されます。
2. **Issueの見積もりが難しい**
「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまいます。

例えば、最近注目されているReactやNextについて。
Reactについては**本リリース後に組み込む方向で検討**するのが良いかと考えています。
その後、React単体のキャッチアップを行い、アプリに組み込む技術的な検証をした上で、組み込むかどうかを判断する形が理想かもしれません。
