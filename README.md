# 1. サービス概要

# MusicHub

クラブやフェスの現場で聴いた音楽を、**イベント単位**で記録できるアプリ。識別できない未リリース曲・Bootleg・Mashupも、曲名が空欄のまま残せる。

曲名がわからなくても、日時・会場・イベント名・DJ名・どの瞬間で曲が流れたか・体感BPM・雰囲気といった現場の文脈と一緒に記録できる。後日、別ルート(動画・知人からの情報・記憶の整理)で「あの曲だった!」と判明した時に、自分の手で記録を更新して識別済みにできる。

---
# 2. このアイデアはどこから生まれたか

## 2-1. きっかけとなった体験・感情

夜22時以降にオープンするクラブハウスでEDMを聴いた時に、お客さんらがサビで盛り上がりDJが現場を盛り上げる熱気に魅了されました。本当に音楽が好きな瞬間にサビのリズムに合わせて手を挙げてジャンプしたり、ヘッドバンギングしたり、メロディに合わせて歌ったりなど、DJのパフォーマンスに合わせて会場が一つになる臨場感・一体感が大好きです。現場で聴いたあの曲をすぐ思い出せるようにできたら、自分の好きな音楽をイベントの思い出と一緒に思い出し、好きな音楽をさらに広げられると考えております。

---
## 2-2. なぜそれが「気になった」のか

私は、興味のある最新トレンドを追っていくのが好きです。また、言葉でなく、音楽や映像など作品を通してクリエイターの気持ちに浸ることが好きです。特に、音楽は高校生の時からずっと聴いてきていて思い入れが強いです。お客さん(オーディエンス)としてだけでなく、DJ/Producerの方にも役立つサービスを届けたいと考えております。

---
# 3. 課題の整理

## 3-1. 表に見えている困りごと

- クラブ会場やフェス会場でDJが次々と曲をかけていくため、いいと感じた曲ごとに記録を付けるのが難しい。
- 現地会場に来た人だけに披露される、未リリース曲やRemix・Bootlegがかかり、既存アプリShazamを使っても解析不能になることがある。
- 後でまとめて見返せる形で残しておきたい。
- 現地会場でお酒を飲んで聴いているため、いちいち記録している余裕がない。

---
## 3-2. 本当に解決したい課題は何か

なぜ?
- 初めて聴いた音楽を忘れぬうちに記録しておきたいから

なぜ?
- 「何これ!?」や「この曲聴いたことがないけどすごく好き!」と感じる新しい音楽を忘れてほしくないから

なぜ?
- 自分の知らない音楽を開拓して、自分の好きを他の人たちに伝えられるようにしたいから

→ 本当に解決したい課題:**Shazamでは識別できず既存サービスのデータベースにも載らない「現場でしか出会えない曲」を、文脈ごと未来の自分に残せるようにする**。曲が特定できた後に共有・開拓に進めるよう、記録の入口を最大限ゆるくする。

# 4. 想定ユーザーについて

## 4-1. 想定しているユーザー

### 年齢設定
20代から40代の音楽が好きで新しい音楽体験を求めている人。  
理由:時間とお金を費やしてもクラブやフェス会場に足を運べて、出会った音楽を通して自分の好きな音楽を掘り下げられる年代と想定しているから。

### 生活・状況設定
音楽イベントで、見たいDJのパフォーマンスを楽しみにしているオーディエンス(一般層)、または、イベント当日に演者としてDJされる方。どちらの場合も本業があり、余暇時間で現地に来られている想定。

コロナ禍以降、円安の影響で週末土曜日になると海外のDJ/Producerをヘッドライナーとするイベントがクラブ会場で多く見られるようになった。イベント当日は、未発表曲含む楽曲を生み出した本人によるDJパフォーマンス、VJやLJによる映像と光の演出、来日DJ/Producerとの写真撮影といった感動や興奮を体験できる。来日DJ/Producerが出演する日はオーディエンスが多く来場するため、演者のDJも盛り上げるために気合いが入っている。イベント会場では人で溢れ、お酒を嗜みながら楽しんでいるため、楽曲一つ一つ覚えている余裕がない。

### 環境とハードル
本当に音楽が好きな人は会場で流れた音楽を後で見返せるように、動画を撮ったり、Shazamで曲解析を試みたり、1001tracklists.comで楽曲をリサーチしたりするが、**「Shazamで識別できなかった曲」と「現場の文脈」を紐付けて残せるプラットフォームがない**。

本webサービスは、曲名空欄でも記録できるようにし、イベント・雰囲気・体感BPMといった文脈情報で「あの夜のあの瞬間」を思い出せるようにする。

---
## 4-2. 自分とユーザーの距離

該当:**身近な誰か(友達)** および **自分自身**。

友達の薦めでクラブミュージックを聴くようになり、音楽体験をまとめられる扱いやすいサービスを作りたかったから。友達と一緒にクラブ会場やフェス会場に行ったり、音楽ゲームをしたりなど、音楽を通して関わりあうことが多い。

## 4-3. 実際に使われる可能性について

### 最初に使ってくれそうな人は誰か
クラブ会場やフェス会場で音楽を聴くのが好きな人、YouTubeやSpotifyでプレイリストを作成して音楽をまとめている人。

### どんな場面でこのアプリを思い出すか
- イベント直後の帰り道、または翌朝、「あの曲なんだっけ」となった時
- 過去体験した音楽を思い出したい時

### 今の生活の中で代わりに使っているもの
- Shazam(識別できた場合のみ機能する)
- YouTubeとSpotifyのプレイリスト(リリース済み楽曲のみ)
- 1001tracklists.com(DJセトリの一方向参照)

### 置き換えてまで使う理由
既存サービスは「曲が特定済み」を前提にしている。Shazamで識別できなかった瞬間に記録手段が失われる。MusicHubは曲名が無くても記録を始められ、後日識別できたら更新できる。**「答えが出るまで待ってくれる」のが他にない価値**。

## 4-4. 利用導入と継続のイメージ

### 知る・思い出すきっかけ
現場で聴いた音楽を覚えておきたいが、Shazamで識別できなかった経験を持つ人。色々な楽曲を知っているが、すぐに思い出せなくて認知負荷がかかっている方。

### 最初の一歩
ランディングページにアクセスし、サインアップ後すぐにイベントを作成し、そのイベントにTrackEntry(曲名空欄可)を記録できる導線にする。

### 利用を続ける理由
- 新しい音楽体験に触れた時のアツい瞬間(臨場感・躍動感)を生きた体験として残せる
- 「未識別エントリ一覧」が**後で答えに行き着くためのTODOリスト**として機能する
- イベントごとに過去の自分の音楽体験が積み重なっていき、自分専用の音楽履歴になる

# 5. 既存サービス・競合調査

## 5-1. 似たサービスの調査

- **MeloLog** — <https://github.com/hinata7777/melo_log>  
Spotifyと連携して、楽曲検索してプレイリストを作成する

- **EMOTION MUSIC BOX (EMOMU)** — <https://github.com/morikkosan/emotion_music_box>  
SoundCloudと連携して、感情で楽曲を一言呟いて投稿・コメント・ブックマークができる

- **Shazam** — <https://www.shazam.com/ja-jp>  
現地会場で流れる音楽を1タップでリアルタイム録音し、何の楽曲か解析してくれる楽曲検索アプリ

- **Spotify** — <https://open.spotify.com/intl-ja>  
1億曲以上の楽曲をストリーミング再生でき、プレイリストを作成できるプラットフォーム

## 5-2. それでも自分が作りたい理由

- **MeloLog**: DJ/Producer名でキーワード検索しても見つからないことがある。Spotify APIでヒットした楽曲のみで、現地で聴いた未リリース曲・Remix・Bootlegは出てこない。

- **EMOMU**: SoundCloud連携を前提としているが、オーディエンスの中にはSoundCloudアカウントを持っていない人がいる。複数プラットフォームを駆使する音楽好きにとって、特定プラットフォーム依存は使いづらい。

- **Shazam**: ドロップ(サビ)の部分で解析するとヒットしやすいが、誤判定や、録音の分解能不足で判定されないまま終わることがある。**識別できなかった時点でサービスとしての役目が終わる**のが最大の限界。

- **Spotify**: リリース済み楽曲はストリーミングできるが、現場でかかる一部のRemix・Bootleg・Mashup・未リリース曲はヒットしない。

**共通する違和感**:既存サービスはすべて「曲が特定済み」を前提にしている。Shazamで識別できなかった瞬間、EMOMUでSoundCloudにない曲だった瞬間、Spotifyに登録されていないBootlegだった瞬間、ユーザーは記録手段を失う。MusicHubは「曲名がわからない状態」を一級市民として扱う点で、これらすべてと異なる。

## 5-3. 差別化を一文で決める

**クラブやフェスの現場で聴いた音楽を、Shazamで識別できない曲も含めて、イベント単位で記録・追跡できるアプリ**。曲名が後から判明したら自分の手で更新でき、識別までの過程ごと音楽体験を残せる。

### 三方向の差別化

| 競合 | 競合の立ち位置 | MusicHub |
|---|---|---|
| Shazam | 単発の曲識別ツール(識別できない時点で終わる) | **識別できない曲も文脈ごと記録できる記憶装置** |
| EMOMU | 楽曲+感情のSNS投稿 | イベント単位の個人ジャーナル(SNS要素なし) |
| Spotify / MeloLog | 既存リリース曲のプレイリスト | 未リリース・Bootleg・空欄エントリ対応 |

## 6-1. ユーザーの変化

- **行動の変化**:Shazamで識別できなかった曲も諦めずに記録に残すようになる

# 7. このアプリで実現すること

## MVPで作る機能(2週間以内リリース目標)

| # | 機能 | Shazam差別化との関係 |
|---|---|---|
| 1 | ユーザーのログイン認証(Rails 8 built-in authentication) | 基盤 |
| 2 | **Eventモデル**のCRUD(日時・会場・イベント名・DJ名) | 基盤 |
| 3 | **TrackEntryモデル**のCRUD ※`title nullable`、`mood_tags`(Postgres array column)、`felt_bpm`、`memo` | ★Shazam差別化の核 |
| 4 | **TrackEntry編集機能**(後から曲名を埋められる) | ★Shazam差別化の核 |
| 5 | **未識別エントリ絞り込み表示**(title is null のフィルタ) | ★Shazam差別化の核 |
| 6 | イベント単位のタイムライン表示(時系列で並ぶ) | 文脈の可視化 |
| 7 | イベント単位の共有リンク発行(read-onlyの公開トークン付きURL) | 後日見返し |

### MVPでのShazam差別化フロー

```
現場でShazamが解析失敗 → MusicHubで曲名空欄エントリ作成
                       (イベント・雰囲気タグ・体感BPM・記憶メモを残す)
                            ↓
                  後日、別ルート(動画、知人、tracklists等)で判明
                            ↓
                  MusicHubで該当エントリを編集して曲名を追記
                            ↓
                  「未識別エントリ一覧」から消えて識別済みに移動
```

このフローを実装することで、Shazamではできない**「答えが出るまで待ってくれる」サービス**になる。

## 本リリースで作る機能

- タグ付け機能の本格化(`acts-as-taggable-on` gem への移行)
- タグ付けによる検索・絞り込み
- プレイリストのCRUD(複数イベントをまたいだTrackEntryのまとめ機能)
- プレイリストの共有リンク発行
- PWA化・オフライン対応(IndexedDB + Solid Queueリトライ)

# 8. このアプリの懸念点とその対策

## 懸念点① 既存サービスとの差別化

**問題**:MeloLog/Spotify/EMOMU/Shazamと表面的な機能が被って見える可能性がある。

**対策**:
- 「Event×TrackEntryのデータモデル」と「曲名nullを許容する記録フロー」をMVP段階から実装し、データモデルレベルで競合と異なることを示す
- Shazamは識別装置・MusicHubは記憶装置という役割分担を明示する(同じ土俵で勝負しない)
- 「未識別エントリ一覧」=後で答えに行き着くTODOリスト、というShazamに存在しないUXを作る

## 懸念点② 業界・ユーザーの解像度不足

**問題**:現場でDJされている方や現場スタッフ、会場に足を運ぶオーディエンスのリサーチが十分にできていない。

**対策**:身近な友人(クラブ通いの仲間)から実際のフィードバックを得ながらMVPを改善。本リリース開発開始後、現地調査も並行する。

## 懸念点③ オフライン環境への対応

**問題**:クラブ会場やフェス会場で通信が不安定になることがある。

**対策**:**MVPでは対象外**とし、「会場を出てから記録する」前提で設計する。本リリースでPWA化・IndexedDB・Solid Queueリトライを段階的に導入。MVPで全部やろうとすると2週間で間に合わないため、優先順位を明確化。

# 9. 今後の展開・発展の方向性


- **タグ付けの強化**:`acts-as-taggable-on`への移行で、ジャンル・雰囲気・場所・時間帯など多軸タグ付けと検索を実現
- **プレイリスト機能**:イベントをまたいだTrackEntryのキュレーション(「夏に聴いたエモい曲集」など)
- **個人の音楽履歴可視化**:データが溜まると、自分のイベント参加履歴・好みの雰囲気・年代別の傾向などをグラフ化できる
- **PWA化によるオフライン対応**:現場でも記録できるようになる

- **SNS共有機能**:ユーザーが自分の音楽履歴やプレイリストをXやInstagramに共有する一方向の連携
- **DJ/Producer向け別サービスとの連携**:DJのセトリ投稿サービスと連携し、同イベント参加者の未識別エントリと突き合わせる(本webサービス本体には組み込まない)

※UIテイストとしてはクラブ・フェスシーンの臨場感を意識したダークモード基調のデザインを採用予定。

## 10-2. 技術選定の理由

### なぜこの技術を使うのか
- なるべく最新バージョンを用いることで、サポート期間を伸ばしてユーザー体験を向上させたい
- 最新パッチをなるべく適用することで既知のセキュリティホールを避けたい(例:npm attack)
- なるべくコストを抑えつつ運用しやすいアーキテクチャにしたい

### 今回チャレンジしたい点
- **Eventモデルを一級市民として設計**し、楽曲記録をイベント単位で管理する
- **Rails 8 built-in authenticationを使った認証実装**(Rails 8新機能の活用)
- **RSpecを活用したTDD**


### 必要なgemの整理

**MVP**
- 認証:Rails 8 built-in authentication(gem追加不要)
- 環境変数:dotenv-rails
- N+1検出:bullet(development)
- 静的解析:rubocop-rails-omakase(Rails 8デフォルト)
- テスト:rspec-rails、factory_bot_rails

**本リリース**
- acts-as-taggable-on(タグ付け本格化)
- Solid Queue(バックグラウンドジョブ)

# 10-3. キャッチアップ不足の懸念
キャッチアップ不足の段階では以下の懸念点があります。

- 完成までのハードルが高い
使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想されます。
外部APIやReactやNext.jsといったFWを別途新規導入するかは、本リリース対応と考えてます。

- Issueの見積もりが難しい
「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまいます。
卒業日から逆算してMVPの提案ができてないため、上記で示したmvp目標よりも下回る可能性があります。

- 業界の理解
　現場でDJされてる方や現場スタッフ、そして会場に足を運んで聴きにくるお客さんのリサーチをして本当に必要なサービスか調査・検証する必要がある。
→これについては、本リリースに向けた開発を始めてから、現地調査もしていく予定です。
