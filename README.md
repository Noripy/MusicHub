## MusicHub
音楽が好きなユーザーがプレイリストを共有し、知った時の音楽を思い出せるようにすることで好きな音楽をさらに広げられるようになれるサービス

音楽ジャンルや雰囲気（明るい・エモい・リラックスするetc）をタグ付けして、パッと見た時に思い出せるようにする

MeloLogでは、DJ/Producer名でキーワード検索しても見つからないことがあった。
Spotify APIを利用してヒットした楽曲のみで、現地で聴いた未リリース曲、RemixやBootlegなどは出てこない。

EMOTION MUSIC BOX (EMOMU)は、SoundCloud連携を前提としているが、オーディエンスの中にはSoundCloudアカウントを持っていない人がいる。
音楽が本当に好きな人たちは、複数のプラットフォーム（感想はXやInstagram、音楽関連はYouTube、Beatport、SoundCloudなど）を駆使して情報を集めに行くため、特定のプラットフォームに依存するのは避けたい。

Shazamは、ドロップ（サビ）の部分で解析かけるとヒットしやすいが、リアルタイム録音された楽曲を誤判定することがあったり、録音した楽曲の分解能が悪く判定されないまま終わることがある。

Spotifyは、世間にリリースされた楽曲はストリーミング再生できる傾向にある一方で、現場でかかる一部のremix、bootleg、mashup、未リリース曲はヒットしない。

# 10-2. 技術選定の理由
- なぜこの技術を使うのか
なるべく最新バージョンを用いることで、サポート期間を伸ばしてユーザーの利用体験を向上させさせたいから。
最新のパッチをなるべく適用することで、既知のセキュリティホールを避けたいから（例. npm attack）。
なるべくコストを抑えつつ、運用しやすいアーキテクチャにしたいから。

- 今回チャレンジしたい点
Deviseを用いた認証基盤構築
Rails7・8の機能を利用したシステム開発
楽曲ごとにタグ付け

※不安な点
- ユーザーAが投稿した楽曲がとユーザーBに既に同じ楽曲が入っている場合のデータベースの関係性の定義がはっきりしていない。
- 技術拡張していくのに必要なgemパッケージの選定が済んでいない。

# 10-3.キャッチアップ不足の懸念
キャッチアップ不足の段階では以下の懸念点があります。

- 完成までのハードルが高い
使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想されます。
外部APIやReactやNext.jsといったFWを新規導入するものは、本リリース対応と考えてます。

- Issueの見積もりが難しい
「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまいます。
卒業日から逆算してMVPの提案ができてないため、上記で示したmvp目標よりも下回る可能性があります。

- 業界の理解
　現場でDJされてる方や現場スタッフ、そして会場に足を運んで聴きにくるお客さんのリサーチをして本当に必要なサービスか調査・検証する必要がある。
→これについては、本リリースに向けた開発を始めてから、現地調査もしていく予定です。
